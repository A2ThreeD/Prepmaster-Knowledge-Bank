# Software Plan For A Fresh Raspberry Pi

This is the current recommended install set for a new Raspberry Pi OS Lite deployment.

## Base Packages Installed By `bootstrap_pi.sh`

- `avahi-daemon` for LAN hostname discovery
- `ca-certificates` for HTTPS downloads
- `curl` and `wget` for scripted downloads
- `git` for upstream repo retrieval
- `hostapd` for wireless access point mode
- `jq` for JSON processing in future automation
- `kiwix-tools` for `kiwix-serve` and `kiwix-manage`
- `nginx` for the landing page and proxy layer
- `python3`, `python3-pip`, `python3-venv` for helper tooling
- `rsync` for file sync operations
- `sqlite3` for lightweight local state if needed
- `dnsmasq` for DHCP/DNS on the local access point
- `ufw` for firewall management
- `unzip` and `xz-utils` for extracted assets

## Software We Still Need To Automate Next

- `kiwix-serve` installation and service configuration
- Offline maps package/data provisioning
- Systemd services for Kiwix and site status collection
- Nginx site configuration for `/`, `/kiwix`, `/maps`, and `/admin`
- A settings workflow for toggling and managing wireless AP mode from the admin experience

## Kiwix Content Source

The curated ZIM library for this project now comes from `kiwix-categories.json`, imported from Project NOMAD.

- `essential` includes essential tiers from each category
- `standard` includes essential and standard tiers
- `comprehensive` includes all tiers

The manifest is generated by `scripts/build_kiwix_zim_manifest.py` into `config/kiwix-zim-urls.txt`.

Wikipedia selection is handled separately through `wikipedia.json` so the user can choose a lighter or heavier Wikipedia package during configuration.

For first-run validation, `PREPMASTER_ZIM_MODE=quick-test` uses `config/kiwix-zim-urls.quick-test.txt` instead of the full generated manifest.

## Likely Runtime Services

- `nginx`
- `avahi-daemon`
- `kiwix-serve`
- `hostapd` when AP mode is enabled
- `dnsmasq` when AP mode is enabled
- `prepmaster-portal.service` for setup state and live status APIs
- Nginx reverse proxy route for `/api`
- Nginx redirect from `/kiwix` to the dedicated Kiwix port

## Browser-Driven Apply Flow

The intended appliance workflow is now:

- run the base installer once
- open the site
- choose options in the setup page
- click `Apply Configuration`
- let the local portal API trigger downloads, optional installs, AP configuration, and service restarts

## First-Start Install Choices

The first startup experience should guide the user through these choices:

- `Base Install` enabled by default: Kiwix, OpenStreetMaps, and supporting system tools
- Optional `Kolibri` add-on for broader offline learning content
- Optional `KA Lite` add-on for legacy Khan Academy-style offline content

Those selections are currently tracked in `config/install-profile.env`.

## Optional Component Automation

- `scripts/install_kolibri.sh` installs Kolibri and enables its system service
- `scripts/install_optional_components.sh` applies the selected profile during the main install
- `scripts/install_ka_lite_legacy.sh` intentionally requires `PREPMASTER_ALLOW_LEGACY_KA_LITE=1`

KA Lite remains a legacy path because its official project status is deprecated and its Linux install expectations are older than current Raspberry Pi OS releases.

## Custom Stack Direction

Prepmaster will own:

- the landing page
- the admin UI
- the install/update scripts
- the maps integration layer
- the wireless AP configuration workflow
- the setup-complete state and dashboard status API
- the local reverse-proxy layout for the full appliance-style site

Kiwix will remain the dedicated offline library engine behind `/kiwix`.
